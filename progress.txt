## Codebase Patterns
- Lineup-aware type contracts live in `lib/types.ts`; use `LineupAwareFitnessConfig.mode` (`fast` or `full`) to distinguish hot GA scoring from UI-ready diagnostics instead of adding another user-selectable fitness algorithm.
- PlayPokemon lineup enumeration lives in `lib/genetic/fitness/lineupEnumeration.ts`; roster input order controls lead iteration, backline pairs are sorted canonically and represented once, and species ids containing the lineup key delimiter (`|`) are invalid.
- Lineup scoring lives in `lib/genetic/fitness/lineupScoring.ts`; use `LineupScoringContext` injection for deterministic tests and `createDefaultLineupScoringContext(...)` when wiring production data helpers.
- Lineup resource path metrics use shield-specific matchup ratings when `LineupScoringContext.getShieldScenarioMatchupRating` is available; missing shield rows are neutral fallbacks and paths are unavailable only when no path-specific shield data exists.
- PlayPokemon roster scoring lives in `lib/genetic/fitness/rosterScoring.ts`; use `PlayPokemonRosterScoringContext.scoreLineup` for cached or deterministic lineup scoring, and only request `lineupScores` through full `LineupAwareFitnessConfig` diagnostics.
- PlayPokemon recommendation helpers live in `lib/genetic/fitness/recommendations.ts`; pass only bounded full-mode lineup diagnostics into `buildPlayPokemonRosterRecommendations(...)` so API-ready output returns recommended lineups and bench utility without exposing all 60 enumerated lineups.
- GBL role-order recommendations use `buildGblLineupRecommendation(...)` from `lib/genetic/fitness/recommendations.ts`; it evaluates all six ordered role permutations with canonical lineup scoring and generation returns the single best recommendation on `Chromosome.recommendedLineups`.
- GA fitness now routes through `evaluatePopulation(population, mode, formatId)` in `lib/genetic/fitness/index.ts`; use `createLineupAwareFitnessContext(...)` for per-run cached full and fast lineup scoring, fast mode during population evaluation, and full roster scoring only for final PlayPokemon diagnostics.
- Frontend generation configuration no longer owns or passes `FitnessAlgorithm`; `TeamManager` sends only format, mode, anchors, and exclusions to `/api/generate-team`, and `TeamGenerator` must not render algorithm-selection UI copy.
- Frontend lineup recommendations should flow from `/api/generate-team` into `TeamManager.generatedTeam`, then render in `AnalysisPanel` with Team Analysis Summary; do not recompute lineup scoring in UI adapters or render recommendation details inside generated-team cards.
- Frontend PlayPokemon roster diagnostics should flow from `/api/generate-team` into `TeamManager.generatedTeam`, then through `ResultsPanel` into `TeamDisplay`; display `rosterMetrics` and `benchUtility` directly and include text warning labels for low-utility or unbringable roster members.
- `GenerationOptions` and `GenerationAnalysis` in `lib/types.ts` no longer include algorithm selectors or labels; adapters should use the canonical lineup-aware generation path and pass through lineup-aware diagnostics.
- Recommended lineup resource path cards should show each path's strategic meaning plus a visible `weak`, `neutral`, `strong`, or `elite` quality label derived from the rounded displayed score; do not rely on score color alone.
- Recommended lineup cards live in `AnalysisPanel` and should stay concise: no covered-threat list, `Lead`/`Switch`/`Closer` terms without repeated labels in values, and weakness ids rendered through analysis names or canonical Pokemon species names.
- `/api/generate-team` should resolve display-facing threat ids in lineup diagnostics, such as recommended lineup weaknesses and roster single-answer risks, to readable species names before returning them so client components stay display-only.
- PlayPokemon roster diagnostics in `TeamDisplay` should render API-provided single-answer risk labels and use `Switch` copy for role appearance counts.
- PlayPokemon roster fitness now combines lineup quality/depth with expected-meta offensive move coverage, defensive resistance/weakness spread using frequency-preserving expected threat move types when available, and a top-lineup primary-type redundancy penalty that includes all lineups tied at the cutoff; use per-Pokemon typing fallback when move metadata is missing or partially unresolved, and handle useful shared secondary typings through coverage context rather than blanket duplicate-type penalties.
- Bench utility warning pills in `TeamDisplay` should use inline flex centering (`inline-flex`, `items-center`, `justify-center`, and `text-center`) so warning labels stay horizontally and vertically centered.
- Recommended lineup weaknesses in `AnalysisPanel` should render as semantic `ul`/`li` lists with one readable weakness label per item and `No major weaknesses identified` for empty weakness arrays; avoid comma-separated inline weakness strings.
