## Codebase Patterns
- Team Analysis Summary UI lives primarily in `components/organisms/AnalysisPanel/AnalysisPanel.tsx`; keep client components display-only and do not recompute optimizer scoring in the UI.
- Recommended lineup diagnostics should flow from `/api/generate-team` into `TeamManager.generatedTeam`, then render in `AnalysisPanel`; do not render recommendation details inside generated-team cards.
- Conditional `AnalysisPanel` accordion sections must derive keyboard focus order from rendered sections so Arrow/Home/End navigation does not focus hidden section refs.
- Accordion headers must use native buttons with `aria-expanded`, `aria-controls`, visible focus styles, and region panels labeled by the corresponding trigger.
- Recommended lineup cards should stay concise: show `Lead`, `Switch`, and `Closer` role terms, avoid repeating those labels inside values, omit numeric lineup score and covered-threat lists, and keep weaknesses as semantic `ul`/`li` lists with one readable weakness label per item.
- Recommended lineup cards should not render `diagnosticLabel`/`Structure`; keep structure labels as internal optimizer/API diagnostics unless a current story explicitly reintroduces a display consumer.
- `/api/generate-team` should resolve display-facing threat and lineup ids to readable species names before returning them so client components stay display-only.
- Keep core optimizer/domain models species-id based. Resolve species names at API/UI adapter boundaries rather than by importing Pokemon data into client analysis components.
- `lib/genetic/fitness/rosterScoring.ts` uses roster metrics internally for `scoreBreakdown` and fitness. Do not delete `buildRosterMetrics` or internal `PlayPokemonRosterScoreResult.metrics` unless scoring is intentionally refactored.
- `Chromosome` results should not expose top-level `rosterMetrics` or `benchUtility`; keep roster metrics inside `scorePlayPokemonRoster(...)` internals unless a current API/UI consumer requires a new explicit adapter field.
- `lib/genetic/fitness/lineupScoring.ts` uses resource path metrics for lineup/roster safety scoring. Remove shield path display fields only from API/UI output unless internals are proven unused.
- `lib/analysis/pokemonContributionAnalysis.ts` returns concise per-Pokemon numeric contribution diagnostics only; do not reintroduce generated rationale text unless a current UI/API consumer explicitly needs display copy.
- `TeamDisplay` should not render PlayPokemon `Roster Metrics`; keep `rosterMetrics` and `benchUtility` as internal optimizer/domain diagnostics unless a current story explicitly reintroduces a display consumer.
- Optimizer score breakdown UI should consume API-provided `scoreBreakdown` through `TeamManager.generatedTeam` and render display-only explanations in `AnalysisPanel`.
- Summary Statistics in `AnalysisPanel` is the optimizer score-card section; do not reintroduce client-computed Overall Fitness, Threat Handling, Shield Stability, Core-Breaker Risk, Expected Ranges, or a separate Optimizer Score Breakdown accordion without a current story.
- Summary Statistics optimizer score cards should render Role immediately after Bulk as a standard card, followed by Offensive Ratio and Defensive Ratio in a dedicated two-card comparison row; do not rely on a single auto-flow grid for the ratio pair.
- Summary Statistics optimizer score cards display only simple `A`, `B`, `C`, `D`, or `F` letter grades; do not render numeric score values, plus/minus modifiers, or `elite`/`strong`/`neutral`/`weak` quality pills there unless a current story changes the contract.
- Summary Statistics may include a separate `Threat Score` diagnostic card sourced from API-provided `scoreBreakdown.threatScore`; keep it lower-is-better, display-only, pill-free, and cap visible Top Meta/Overall Team Threat lists with a count summary.
- Recommended Lineups belongs in the `AnalysisPanel` accordion immediately after Summary Statistics; keep it in the rendered-section list so keyboard navigation follows actual visible section order.
- Recommended Lineups accordion cards use blue diagnostic styling consistent with Summary Statistics and the surrounding analysis accordion; do not reintroduce emerald/green card styling unless a current story explicitly changes it.
- Recommended Lineup cards render exactly one textual quality pill (`elite`, `strong`, `neutral`, or `weak`) from API-provided lineup `score` metadata; keep numeric score values hidden and do not reuse these pills in Summary Statistics.
- Per-Pokemon Contribution headers should render Replacement Risk in a dedicated row below the Pokemon name; keep the metric boxes unchanged below the header.
- For optimizer-adjacent changes, invoke/read the `gbl-optimizer` skill and record optimizer docs read in this file before marking the story complete.
- PlayPokemon bring-6 lineup enumeration evaluates 120 ordered lead/switch/closer assignments (`6 * 5 * 4`); do not canonicalize or sort switch/closer backline pairs because role order changes scoring.
- Final generated `Chromosome.fitness` should match the attached final `scoreBreakdown.score`; PlayPokemon finalists use recomputed full roster fitness and GBL finalists use the final lineup recommendation score before multi-team sorting.
- Specialist delegation guidance: use `@api-designer` for `/api/generate-team` response contract changes, `@backend-developer` for domain deletion-boundary reviews, `@frontend-developer` for React/component/test implementation, and `@ui-designer` for accordion/card hierarchy and visual QA.
- Runtime ranking data includes `chargers`, `attackers`, and `consistency` CSVs for supported formats; ranking tests should assert these role-support categories load instead of treating them as optional missing files.
- Runtime app, component, and optimizer code must not import, require, bundle, or load PvPoke vendor JavaScript; keep local PvPoke engine execution isolated to sync tooling and use `lib/architecture/pvpokeRuntimeBoundary.test.ts` for boundary regressions.
- Optimizer Threat Score diagnostics live on `OptimizerScoreBreakdown.threatScore` and are lower-is-better display/diagnostic data; do not add Threat Score to the weighted fitness components unless a current story explicitly changes optimizer selection behavior, and keep it disabled in fast/GA hot paths.
- Threat Score aggregation uses normalized active `pools.topMeta` and `pools.fullMeta` diagnostics; keep top-meta weighted higher by default, pass season-specific overrides through `LineupScoringContext.threatScorePoolWeights`, fall back to defaults for full-meta-dominant overrides, normalize configured weights over evaluated pools only, and let full-meta contribute without overpowering top-meta.
- Soft matchup scoring is centralized in `lib/genetic/fitness/matchupScoring.ts`; use `scoreMatchupRating(...)` for optimizer scoring/diagnostic quality where close ratings should be intermediate, while preserving binary threshold counts for categorical labels such as covered threats, weaknesses, dominating matchups, and overwhelming losses.
- Lineup-aware fitness caches are per-run context caches in `lib/genetic/fitness/index.ts`; keep keys versioned and format-scoped, and validate cache effectiveness with `cacheStats` counters instead of timing-only assertions.
- Great League Show-6 Pick-3 calibration fixtures live in `data/calibration/great-league-show6-pick3.json` and load through `lib/data/calibrationFixtures.ts`; keep them tournament-backed, calibration-only, and covered by broad invariant tests rather than exact optimizer score or winner snapshots.
- Optimizer docs under `docs/` should mirror the current diagnostics contract: PvPoke vendor JS is reference-only and blocked at runtime, Threat Score is lower-is-better and diagnostic-only, Summary Statistics uses A-F grades without quality pills, and lineup quality pills belong only on Recommended Lineups cards.
